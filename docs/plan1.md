建议：**不要新建一个完全独立的测试项目，也不要一开始就把完整通话业务写进现有主流程。**

最稳的做法是：**在现有程序里做一个隔离的 RTC 测试入口 / 测试窗口**。

原因很明确：

1. RTC 必须最终跑在 Tauri 环境里  
阿里云 RTC Web SDK 涉及麦克风权限、WebView、音频播放、HTTPS/本地环境、设备枚举。新建普通网页测试项目即使跑通，也不能完全代表 Tauri 里的真实表现。

2. 但不能直接接入正式聊天业务  
现在还只是验证“RTC 能不能通”。如果直接做成正式语音通话，会把来电、接听、挂断、通话状态、call_session、群抢麦这些业务复杂度一起引进来，排查问题会很痛苦。

3. 现有程序已有账号、聊天窗口、后端代理  
后续 RTC token 权限要绑定当前用户、单聊会话、群会话。放在现有程序里测试，后面迁移成本最低。

**推荐方案**
在现有程序里加一个“隐藏/开发用 RTC 测试面板”，例如：
- 不出现在普通用户主流程里。
- 可从开发按钮、临时菜单、或固定测试入口打开。
- 手动输入 `channelId`。
- 当前登录用户自动作为 `userId`。
- 按钮只做：
  - 获取 token
  - 加入频道
  - 开麦/关麦
  - 离开频道
  - 显示远端用户加入/离开
  - 显示错误日志

**后端也不建议另起一个长期独立测试服务**
`plan4.md` 里的独立 `/opt/teacher-rtc-test` 服务可以作为最初验证 AppID/AppKey 生成 token 的临时脚手架，但正式开发时应尽快并入当前后端：
- `/rtc/token`
- 使用现有登录鉴权
- 不把 AppKey 暴露给前端
- 校验用户是否能进入指定会话/群

**结论**
下一步建议做：  
**现有项目内的 RTC 基础设施测试模式**，不是新建独立项目。  
先完成“两台客户端 / 两个账号进入同一个测试频道能听到声音”，再进入普通 1v1 电话业务。
