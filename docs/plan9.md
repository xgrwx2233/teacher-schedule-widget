下面是基于当前项目的班级账号 / 班级群功能设计 plan，不写代码。

**总体建议**
- 现阶段不要单独拆“班级端”项目。先在现有 IM 体系里增加 `账号类型` 和 `群类型`，让班级账号、班级群复用普通用户 / 普通群的聊天能力。
- 等聊天通信模块稳定后，再把“仅聊天模块”抽成班级端。这符合标准 IM 的渐进做法：先统一账号、关系、会话、消息模型，再按端裁剪功能。
- 班级账号本质上应是一个特殊用户账号，班级群本质上应是一个特殊群。不要另起一套平行聊天系统，否则后续好友、搜索、消息、文件、群权限都会重复开发。

**数据模型**
- `users` 增加账号类型：`account_type = normal | class`。
- 班级账号仍占用一条 `users` 记录，可以登录、收发消息、进入会话。
- `user_profiles` 继续保存昵称、头像、简介；班级账号也复用这张表。
- 新增班级资料表，例如 `class_accounts`：保存 `class_no`、`user_id`、`owner_user_id`、`linked_phone`、审核状态、创建时间。
- `class_no` 设计为全数字、系统唯一、不可修改。建议 10 位或 11 位，例如 `8 + 9位随机/序列校验码`，避免和手机号混淆。示例：`8604829137`。

- `chat_groups` 增加群类型：`group_type = normal | class`。当前已有 `group_no`，班级群也继续使用群号。
- `friend_relations` 不需要另建班级好友表，班级账号作为 user 直接加入好友关系。

**班级账号申请流程**
- 聊天中间栏右上角 `+` 菜单新增：`申请班级账号`、`创建班级群`。
- 点击 `申请班级账号` 打开班级资料编辑窗口，复用个人资料编辑窗口的结构。
- 字段包括：班级头像、班级昵称、班级简介、关联手机号。
- 关联手机号默认填当前登录用户手机号，但允许修改。
- 点击确定后后端创建一个 `account_type=class` 的用户、生成唯一 `class_no`、写入班级资料。
- 后端自动建立申请人与班级账号的双向好友关系。
- 后端自动创建或获取申请人与班级账号的一对一会话。
- 前端申请成功后关闭窗口，刷新好友 / 会话列表，并进入该会话。

**班级账号登录**
- 增加登录模式：`班级号 + 验证码`。
- 输入班级号后，后端根据 `class_accounts.class_no` 找到关联手机号，向该手机号发送验证码。
- 验证码校验成功后，签发 class user 的 token。
- 登录后当前账号身份就是班级账号，因此聊天里的 sender、好友关系、群成员身份都按班级账号处理。
- 扫码登录先预留数据结构和入口，不进入第一阶段开发。

**创建班级群**
- 点击 `创建班级群` 后复用当前创建群聊的选人窗口。
- 区别是成员列表要展示账号分类：普通账号、班级账号。
- 创建班级群时必须至少选择一个班级账号，否则按钮置灰或提交时报错。
- 后端创建群时传 `group_type=class`，并校验成员里至少包含一个 `account_type=class` 的好友。
- 班级群创建成功后，进入群会话。
- 普通群创建逻辑不变，只是默认 `group_type=normal`。

**搜索与添加**
- 综合搜索升级为搜索用户、班级账号、普通群、班级群。
- 搜索字段包括：手机号、普通昵称、班级号、班级昵称、普通群昵称、班级群昵称、群号。
- 搜索结果 UI 增加分类标签，例如 `用户`、`班级`、`群聊`、`班级群`。
- 添加班级账号为好友的流程与普通用户一致。
- 申请加入班级群的流程与普通群一致。
- 唯一区别是展示标识和文案更清晰，例如“添加班级账号”“申请加入班级群”。

**UI 标识**
- 班级账号昵称旁加小图标，例如蓝色 `班` badge 或小楼/班牌图标。
- 班级群昵称旁加 `班级群` badge。
- 会话列表、联系人列表、搜索结果、群成员列表、个人/班级资料页都应统一显示这个标识。
- 不建议只靠头像区分，标准 IM 通常会用账号类型 badge 明确表达身份。

**接口规划**
- `POST /class-accounts`：申请班级账号。
- `GET /class-accounts/{class_no}`：查询班级账号资料。
- `POST /auth/class/request_code`：班级号获取验证码。
- `POST /auth/class/login_code`：班级号验证码登录。
- `POST /chat/groups` 增加 `groupType` 参数。
- `GET /profile/search` 扩展返回字段：用户类型、班级号、群类型、群号。
- Tauri command 层跟随新增：申请班级账号、打开班级资料编辑窗口、班级登录、创建班级群。

**开发阶段**
- 第一阶段：数据模型与接口。新增账号类型、班级号、群类型，完成班级账号申请、自动加好友、自动建会话。
- 第二阶段：前端入口与资料窗口。加号菜单、班级资料编辑窗口、申请成功跳转会话。
- 第三阶段：创建班级群。复用建群窗口，加账号分类与“必须选择班级账号”校验。
- 第四阶段：综合搜索升级。支持班级号、班级昵称、班级群昵称、群号搜索。
- 第五阶段：班级账号登录。先做班级号验证码登录，扫码登录留接口占位。
- 第六阶段：班级端拆分准备。等聊天模块稳定后，用同一套 IM 代码按账号类型 / 启动入口隐藏课程表功能。

**关键原则**
- 班级账号是 `特殊用户`，不是新聊天实体。
- 班级群是 `特殊群`，不是新群系统。
- 班级端先不拆项目，先在同一工程内通过账号类型裁剪能力。
- 所有聊天能力继续复用现有会话、消息、好友、群成员、群申请、文件链路。
