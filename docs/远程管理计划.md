# 教师助手远程管理与聊天面板开发计划

本计划基于现有 `teacher-schedule-widget` 桌面端项目、`teacher_scheduleV2` 移动端与后端项目，以及 UI 参考图和批注重新整理。当前阶段目标是把课程表小挂件升级为“教师助手”，先完成 Windows Tauri 桌面端聊天面板入口和 mock UI 闭环，再逐步接入后端 IM、WebSocket、OSS、RTC 等能力。

## 1. 当前结论

- 课程表小挂件继续保留为教师助手中的课程表模块。
- 只有课程表小挂件在 attach 模式下作为壁纸小挂件存在。
- 聊天面板、登录、设置、好友资料、好友通知、群资料、壁纸管理等其他窗口均为普通窗口。
- 聊天面板使用自建标题栏，但窗口类型仍然是普通桌面窗口。
- 第一版允许先做 mock UI + 入口闭环，再接真实后端。
- 本阶段不做“远程控制电脑”功能，也不做远程桌面、屏幕共享、鼠标键盘控制。
- 顶部聊天操作保留“对讲”“远程壁纸管理”“远程关闭电脑”“更多”，删除“收藏”和“远程控制电脑”。

## 4. “系统状态气泡”说明

系统状态气泡是聊天时间线中由系统生成的中性提示，不是某个用户发送的普通聊天消息，也不是远程管理状态消息。

典型特征：

- 通常居中显示。
- 没有头像。
- 视觉上比普通消息更弱，常用浅灰色或浅色提示条。
- 用于解释聊天上下文变化，而不是承载业务操作流程。

第一版可保留的例子：

- “今天 10:28”
- “你撤回了一条消息”
- “对方撤回了一条消息”
- “已添加为好友，现在可以开始聊天”
- “群成员已加入群聊”

第一版不做的例子：

- “已发送远程控制请求”
- “正在查看对方屏幕”
- “远程控制已建立”
- “远程控制会话已结束”

这些属于远程管理状态消息，已按批注从第一版中删除。

## 6. 第四阶段：协议与数据模型固化

在桌面端接真实后端之前，先定义统一协议，避免桌面端和移动端再次出现字段、状态、同步规则不一致。

建议新增或整理协议文档：

```text
docs/protocol/
  - chat-overview.md
  - conversation.md
  - message.md
  - websocket-events.md
  - talk-room.md
  - wallpaper-management.md
  - device-command.md
```

核心模型：

- `conversation`
- `message`
- `friend`
- `group`
- `group_member`
- `talk_room`
- `file_object`
- `wallpaper_task`
- `device_command`

消息状态：

- `local_created`
- `sending`
- `sent`
- `delivered`
- `read`
- `failed`
- `revoked`

WebSocket 事件：

- `message.send`
- `message.ack`
- `message.new`
- `message.read`
- `message.revoke`
- `conversation.sync`
- `conversation.unread_changed`
- `friend.requested`
- `friend.accepted`
- `group.created`
- `group.member_added`
- `talk.room_created`
- `talk.room_joined`
- `talk.room_left`
- `talk.room_closed`

- `wallpaper.task_finished`
- `device.shutdown_requested`
- `device.shutdown_accepted`

## 8. 第六阶段：桌面端真实数据接入

前端新增模块建议：

```text
src/features/chat/
  - types.ts
  - mockChat.ts
  - chatRepository.ts
  - chatService.ts
  - chatStore.ts
  - websocketClient.ts
```

接入顺序：

1. 使用 HTTP 拉取会话列表。
2. 使用 HTTP 拉取当前会话历史消息。
3. 发送单聊文本消息。
4. WebSocket 接收新消息。
5. WebSocket 接收已读、撤回、未读数变化。
6. 接入好友资料。
7. 接入群聊。
8. 接入图片和文件。
9. 接入本地缓存。

本地缓存建议：

- 桌面端后续使用 SQLite。
- 移动端继续使用 SQLite / Drift 或现有可维护方案。
- 两端缓存表结构尽量一致。
- 不要每次打开聊天面板都全量拉取历史消息。

2. 对齐数据模型。
3. 对齐 HTTP API。
4. 对齐 WebSocket 事件。
5. 实现会话列表。
6. 实现单聊。
7. 实现群聊。
8. 实现对讲。
9. 实现壁纸任务确认。
10. 实现远程关闭电脑确认。

Flutter 端 RTC 建议：

- Dart 层定义统一 `RtcService`。
- Android 侧通过 MethodChannel 调用阿里云 RTC Android SDK。
- 不把 RTC SDK 调用直接写在页面组件中。
