# 七、API 和数据模型必须统一生成

你要避免两个端手写两套接口。

建议这样做：

## 后端提供 OpenAPI

例如：

```text
/openapi.yaml
```

然后自动生成：

```text
Tauri 前端 TypeScript API Client
Flutter Dart API Client
```

这样两个端调用的接口字段天然一致。

例如统一消息模型：

```json
{
  "id": "msg_10001",
  "conversationId": "conv_10001",
  "senderId": "u_10001",
  "messageType": "text",
  "content": "你好",
  "clientMsgId": "client_uuid",
  "serverSeq": 1024,
  "status": "sent",
  "createdAt": "2026-06-18T10:00:00Z"
}
```

不要让 Flutter 自己写一个：

```dart
class ChatMessage {}
```

Tauri 再写一个：

```ts
interface Message {}
```

然后字段还不一样。

正确做法是：

```text
后端 schema 是唯一真相
↓
自动生成 TypeScript 类型
↓
自动生成 Dart 类型
↓
两端只消费生成结果
```

# 九、群对讲的统一流程

阿里云 RTC 不应该直接散落在两个端的业务代码里。

你应该封装成统一的 `RtcService`。

## 服务端流程

```text
老师点击“开始对讲”
  ↓
调用后端 API：创建 talk_room
  ↓
后端校验老师权限
  ↓
后端创建 RTC channelId
  ↓
后端生成 RTC Token
  ↓
后端通过 WebSocket 通知群成员
  ↓
客户端加入 RTC 房间
```

## 客户端流程

```text
收到 talk.room_created
  ↓
展示“老师发起群对讲”
  ↓
用户点击加入
  ↓
调用 /talk-room/{id}/join
  ↓
后端返回 RTC 参数
  ↓
客户端调用阿里云 RTC SDK 加入频道
```

不要让客户端自己生成 RTC Token。阿里云 RTC 的 Token 是安全签名，应由服务端生成并下发，避免密钥泄露；阿里云开发者文档也说明 Token 用于防止恶意盗用云服务流量。([阿里云开发者社区][3])

# 十、Tauri 端和 Flutter 端如何保持一致

你需要做四个统一。

## 1. 统一接口

```text

```

## 2. 统一状态机

聊天消息状态必须一致：

```text

```

对讲房间状态必须一致：


WebSocket 状态必须一致：

```text

```

## 3. 统一本地缓存结构

Flutter 可以用 SQLite / Drift。
Tauri 可以用 SQLite。

两个端本地缓存表结构尽量一致：

```text

local_message

local_sync_cursor
```

尤其要有：

```text

```

不要每次启动都全量拉数据。

## 4. 统一同步机制

客户端启动时：

```text

  ↓

  ↓
提交本地 lastSeq
  ↓
服务端返回增量消息
  ↓
更新本地缓存
  ↓
更新 UI
```

建议以 `serverSeq` 做消息同步基准。

# 十二、Tauri 接入阿里云 RTC 的建议

Tauri Windows 端有两种方案。

## 方案 A：Tauri 前端直接用阿里云 ARTC Web SDK

优点：

```text
1. 前端接入快
2. TypeScript 代码好维护
3. 和聊天 UI 在同一层
4. 不需要马上写 Rust FFI
5. 适合第一阶段纯音频群对讲
```

缺点：

```text
1. 要重点测试 WebView2 下麦克风权限
2. 音频设备切换能力可能不如原生 SDK
3. 对底层音频控制能力有限
```

阿里云 ARTC Web SDK 提供 `AliRtcEngine`，包含加入频道、离开频道、获取麦克风设备、订阅远端音频等能力。([阿里云][4])

## 方案 B：Tauri Rust 插件接 Windows 原生 SDK

优点：

```text
1. 稳定性和音频设备控制更强
2. 更适合长期做桌面通信软件
3. 后续如果做屏幕共享、系统级能力更方便
```

缺点：

```text
1. 开发复杂
2. Codex 修改难度高
3. Rust / C++ / SDK 桥接工作量大
4. 调试成本高
```

我的建议：

```text
第一版：Tauri 使用 ARTC Web SDK
第二版：如果 WebView2 音频兼容性不够，再换 Windows 原生 SDK
```

# 十三、Flutter Android 接入阿里云 RTC 的建议

阿里云官方 ARTC SDK 下载页明确提供 Android SDK 和 Android 集成方式，包括 Maven 集成、权限配置、混淆规则等；Android 端需要处理录音、网络状态、音频设置、蓝牙等权限。([阿里云][1])

但你用的是 Flutter，所以建议：

```text
Flutter UI / 状态管理
  ↓
Dart RtcService 接口
  ↓
MethodChannel
  ↓
Android 原生 Kotlin/Java
  ↓
阿里云 ARTC Android SDK
```

也就是 Flutter 不要直接到处调用原生 SDK，而是封装成：

```dart
abstract class RtcService {
  Future<void> joinTalkRoom(RtcJoinParams params);
  Future<void> leaveTalkRoom();
  Future<void> muteMicrophone(bool muted);
  Stream<RtcEvent> get events;
}
```

Tauri 端也做类似接口：

```ts
interface RtcService {
  joinTalkRoom(params: RtcJoinParams): Promise<void>;
  leaveTalkRoom(): Promise<void>;
  muteMicrophone(muted: boolean): Promise<void>;
  events: EventEmitter<RtcEvent>;
}
```

这样两端业务代码的思路是一致的。


## 错误 2：两个端各自封装接口

不能 Flutter 一套 API，Tauri 一套 API。

必须：

```text
OpenAPI 统一生成
```

## 错误 3：RTC 逻辑写进页面

不要在某个页面里直接写阿里云 RTC 加入房间逻辑。

应该封装：

```text
RtcService
TalkRoomRepository
TalkRoomController
```

## 错误 4：消息没有 serverSeq

没有统一递增序号，后期同步会非常痛苦。

必须有：

```text
serverSeq
conversationSeq
syncCursor
```

## 错误 5：不做本地缓存

聊天类软件不能每次打开都重新请求全部消息。
桌面端和移动端都要有本地缓存。

---

# 二十、最终建议

你的开发策略应该是：

```text
不要桌面端和移动端同时平行开发。
先统一后端、协议、数据库、同步机制。
然后先做 Windows Tauri 桌面端完整闭环。
再用同一套协议和生成代码做 Flutter Android 端。
```

一句话版本：

> **底层统一，端上分步；协议先行，桌面先闭环，移动端后对齐。**

最适合你的当前架构是：

```text
Tauri Windows + Flutter Android
        ↓
统一 OpenAPI / WebSocket 协议 / RTC 房间协议
        ↓
ECS 后端服务
        ↓
RDS MySQL + Redis + OSS
        ↓
阿里云 RTC 群对讲
```

这样你后面再加远程控制功能时，也不会把现有课程表、聊天、群聊、对讲全部推翻。
